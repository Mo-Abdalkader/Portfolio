LSTM & GRU:
Teaching Networks to Remember

In Day 20 we derived exactly why RNN gradients vanish — the gradient at step 1 involves multiplying W_h by itself T−1 times, and it decays exponentially to zero. The LSTM (1997) solved this with one elegant idea: a separate memory cell that information can flow through with almost no modification. The GRU (2014) distilled the same insight into a simpler two-gate design. Both remain in production today. This post builds both from scratch, derives the gradient highway mathematically, and shows exactly where the transformer takes over.

← Back to Blog ☰ Categories

The vanishing gradient is not a numerical accident.
It is a theorem: multiply any matrix with spectral radius < 1 by itself
enough times and the result approaches zero.

The LSTM's answer was architectural, not numerical:
stop multiplying. Give memory a path that adds instead.

01 — A Precise Statement of the Problem

Before building the solution, let's state the problem with full precision. From Day 20, the gradient of the loss at step T with respect to the hidden state at step 1 is:

∂L/∂h_1 = ∂L/∂h_T · ∏(t=2 to T) ∂h_t/∂h_{t-1}

Each Jacobian: ∂h_t/∂h_{t-1} = diag(sech²(·)) · W_h

Problem: this product involves W_h multiplied by itself (T−1) times.
If ||W_h||_2 < 1 → product shrinks exponentially → vanishing gradient
If ||W_h||_2 > 1 → product grows exponentially → exploding gradient

The root cause is the multiplicative recurrence: to update the hidden state, we always multiply by W_h. There is no additive path — every gradient step must pass through this multiplication. For the RNN, the only connection between h_1 and h_T is through this chain of multiplications.

The architectural insight behind both LSTM and GRU is the same:

💡 The Core Insight

Replace the multiplicative recurrence with an additive one. Instead of h_t = f(W_h · h_{t-1} + ...), create a memory variable C_t where the update is C_t = C_{t-1} + something. Gradients flowing back through addition don't decay — ∂C_t/∂C_{t-1} = 1 regardless of how many steps ago. This is the gradient highway.

The gates (forget, input, output) are the mechanisms that control what flows through this highway and when. But the highway itself — the additive cell state — is the fundamental innovation.

02 — LSTM: Architecture from First Principles

The Long Short-Term Memory was introduced by Hochreiter & Schmidhuber in 1997 — one of the most cited papers in deep learning history. It introduces two running states instead of one:

Cell state C_t
The "long-term memory." A vector of the same size as the hidden state that travels through time with minimal transformation — the gradient highway. Information can be stored in C_t for hundreds of timesteps without decaying.
Hidden state h_t
The "working memory." Derived from the cell state via the output gate — this is what gets passed to the next layer or used to compute predictions. It's a filtered, compressed version of C_t.

Three gates control the flow of information. Each gate produces a vector of values between 0 and 1 (via sigmoid), acting as a soft switch: 0 = block completely, 1 = pass through completely, 0.7 = pass through 70%.

The Four Equations

Inputs: x_t (current token), h_{t-1} (previous hidden), C_{t-1} (previous cell)
Notation: [h_{t-1}, x_t] means concatenation of both vectors

① Forget Gate:    f_t = σ( W_f · [h_{t-1}, x_t] + b_f )
   → What fraction of the old cell state to keep? (0=erase all, 1=keep all)

② Input Gate:     i_t = σ( W_i · [h_{t-1}, x_t] + b_i )
③ Candidate:      g_t = tanh( W_g · [h_{t-1}, x_t] + b_g )
   → What new information to add? i_t decides how much, g_t decides what.

④ Cell State Update: C_t = f_t ⊙ C_{t-1} + i_t ⊙ g_t
   → The gradient highway: additive update, no matrix multiplication on C

⑤ Output Gate:    o_t = σ( W_o · [h_{t-1}, x_t] + b_o )
⑥ Hidden State:   h_t = o_t ⊙ tanh( C_t )
   → Filter the cell state to produce the output representation

⊙ = element-wise (Hadamard) multiplication    σ = sigmoid    tanh = tanh
🔍 Why These Specific Operations?

Sigmoid for gates: sigmoid outputs (0, 1), which is exactly what you want from a "how much?" question. 0 = completely closed, 1 = completely open, 0.6 = 60% open. Using tanh would give (−1, 1) — negative fractions don't make sense for a gate.

tanh for candidate g_t: the new information being written to the cell state should be bounded and zero-centred — tanh's (−1, 1) range is correct here. Writing unbounded values into C_t would cause it to grow without limit over many steps.

Addition in the cell update: this is the crucial choice. C_t = f_t ⊙ C_{t-1} + i_t ⊙ g_t uses addition, not multiplication by a weight matrix. The gradient ∂C_t/∂C_{t-1} = f_t (element-wise) — it's just the forget gate value, typically close to 1, not a matrix multiplication that can shrink exponentially.

The Gradient Highway: Mathematical Proof

Let's verify exactly why the cell state fixes the vanishing gradient. The gradient of C_T with respect to C_1:

∂C_T/∂C_1 = ∏(t=2 to T) ∂C_t/∂C_{t-1} = ∏(t=2 to T) f_t

Compare to RNN: ∂h_T/∂h_1 = ∏(t=2 to T) diag(sech²(·)) · W_h

LSTM cell gradient: product of forget gate values (scalars near 1)
RNN hidden gradient: product of (tanh derivative × W_h) — decays exponentially

If f_t ≈ 1 for all t: gradient flows back unchanged across all T steps.
The network learns f_t — it can keep gates near 1 for memories it wants to preserve.

The forget gate is learned, not fixed. When the network encounters information it wants to remember for a long time — a subject at the start of a sentence that needs to match a pronoun 40 words later — it learns to set f_t ≈ 1 for those dimensions of C across those steps. The gradient path stays open exactly where it needs to be.

04 — Worked Forward Pass: "I love NLP"

Let's trace one LSTM step manually. hidden_size = 1 (scalar for clarity). All weights = 0.5, all biases = 0. Previous state: h_0 = 0, C_0 = 0. Input: x_1 = 1 ("I").

── Step 1: x_1="I"=1, h_0=0, C_0=0 ────────────────────────────
z = [h_0 ; x_1] = [0, 1]    (concat)

f_1 = σ(0.5×0 + 0.5×1) = σ(0.5) = 0.622    ← forget gate
i_1 = σ(0.5×0 + 0.5×1) = σ(0.5) = 0.622    ← input gate
g_1 = tanh(0.5×0 + 0.5×1) = tanh(0.5) = 0.462  ← candidate
o_1 = σ(0.5×0 + 0.5×1) = σ(0.5) = 0.622    ← output gate

C_1 = f_1 × C_0 + i_1 × g_1 = 0.622×0 + 0.622×0.462 = 0.287
h_1 = o_1 × tanh(C_1) = 0.622 × tanh(0.287) = 0.622 × 0.280 = 0.174

── Step 2: x_2="love"=1, h_1=0.174, C_1=0.287 ─────────────
z = [0.174, 1]

f_2 = σ(0.5×0.174 + 0.5×1) = σ(0.587) = 0.643
i_2 = σ(0.5×0.174 + 0.5×1) = σ(0.587) = 0.643
g_2 = tanh(0.587) = 0.528
o_2 = σ(0.587) = 0.643

C_2 = 0.643×0.287 + 0.643×0.528 = 0.184 + 0.340 = 0.524
     ↑ C_1 still alive in C_2 — memory preserved
h_2 = 0.643 × tanh(0.524) = 0.643 × 0.481 = 0.309

🔍 What Just Happened

Notice that C_1 = 0.287 contributed 0.184 to C_2 — it was multiplied by f_2 = 0.643, not by a full weight matrix. In a trained model, f_t can be driven to near 1.0 for important dimensions, meaning C_1's contribution to C_100 could still be 0.287 × 0.99^99 ≈ 0.244 — a meaningful signal, not near zero. Compare to the RNN where h_1's contribution to h_100 would involve W_h^99, which for typical random initialisation reaches ~10^-7.

05 — What Each Gate Learns in Practice

After training on real data, the four gates develop interpretable specialisations. Here's what mechanistic interpretability research and practical experience have found:

Forget Gate f_t
Learns to reset context at sentence boundaries, paragraph breaks, or topic changes. When the model encounters a full stop or a discourse marker like "However," f_t drops toward 0 for most dimensions — wiping the cell state clean so the next sentence starts fresh. For within-sentence memory (pronoun resolution), f_t stays near 1.
Input Gate i_t
Controls how much the current token updates the memory. High i_t for semantically rich words (nouns, verbs, named entities), low i_t for function words ("the", "of", "and") that carry little new meaning. The model learns that "the" rarely needs to be stored in long-term memory.
Candidate g_t
Encodes what the new information actually is — the content to potentially write into the cell. Works jointly with i_t: i_t decides how much of g_t to write. Together they implement a gated write: new_info = i_t ⊙ g_t. The candidate is the proposed update; the input gate is the approval mechanism.
Output Gate o_t
Decides what the hidden state exposes to the next layer. The cell state C_t is the full long-term memory; h_t is a selective readout. For translation, o_t might expose "the subject is plural" when the decoder needs to generate a plural verb, while keeping the rest of C_t private. Acts as a context-sensitive filter on top of the full memory.

06 — LSTM Variants Worth Knowing

Peephole Connections (Gers & Schmidhuber, 2000)

In the standard LSTM, gates only see h_{t-1} and x_t. Peephole connections let the gates also see the cell state C_{t-1} directly:

f_t = σ( W_f · [h_{t-1}, x_t] + W_fp · C_{t-1} + b_f )
i_t = σ( W_i · [h_{t-1}, x_t] + W_ip · C_{t-1} + b_i )
o_t = σ( W_o · [h_{t-1}, x_t] + W_op · C_t    + b_o )

W_fp, W_ip, W_op are diagonal matrices (one weight per cell dimension)

Peepholes let gates "know" what's already stored in memory before deciding whether to forget or write. Useful for tasks requiring precise timing — counting exact beats in music generation, or matching opening/closing brackets in code. In practice, the improvement is small and peepholes are rarely used today.

Coupled Forget and Input Gates

A simpler variant that ties f_t and i_t together: i_t = 1 − f_t. If you forget 60% of the old state, you write 40% of the new state. This halves the number of gate parameters and works well on many tasks. The GRU takes this coupling idea further.

Bidirectional LSTM (BiLSTM)

Two LSTMs run in opposite directions over the same sequence — one left-to-right (forward), one right-to-left (backward). Their hidden states are concatenated at each timestep: h_t = [h_t_fwd ; h_t_bwd]. Every token now has access to both past and future context. BiLSTMs were state-of-the-art for NER, POS tagging, and coreference resolution from ~2016–2018, before being replaced by BERT.

Stacked (Deep) LSTM

Multiple LSTM layers stacked vertically — the h_t sequence from layer k becomes the x_t input sequence for layer k+1. GPT-2's predecessor used a 2-layer LSTM for language modelling. Google's Neural Machine Translation (GNMT) used an 8-layer encoder-decoder LSTM in 2016. Depth provides hierarchical representation: lower layers capture local syntax, higher layers capture longer-range semantics.

07 — GRU: The Streamlined Alternative

The Gated Recurrent Unit (Cho et al., 2014) was introduced in the same paper as the seq2seq architecture. Cho's observation: the LSTM has three gates and two states (h and C). This is more complexity than necessary. Can you get the same gradient flow benefits with fewer parameters?

The GRU collapses the LSTM's four equations into two gates and one state:

Inputs: x_t (current token), h_{t-1} (previous hidden — only one state, no C)

① Reset Gate:    r_t = σ( W_r · [h_{t-1}, x_t] + b_r )
   → How much of h_{t-1} to use when computing the candidate? (0=ignore past)

② Update Gate:   z_t = σ( W_z · [h_{t-1}, x_t] + b_z )
   → How much to update the hidden state? (1=full update, 0=carry forward)

③ Candidate:     ñ_t = tanh( W_n · [r_t ⊙ h_{t-1}, x_t] + b_n )
   → Proposed new hidden state, with reset gate filtering past memory

④ New Hidden:    h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ ñ_t
   → Interpolate between old state (carry forward) and new candidate (update)

GRU Cell Diagram

How GRU Achieves the Gradient Highway

The GRU's gradient path is through the update gate equation. When z_t ≈ 0:

h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ ñ_t
If z_t ≈ 0:   h_t ≈ h_{t-1}    (carry forward, no update)
∂h_t/∂h_{t-1} ≈ (1 − z_t) ≈ 1     (gradient passes through unchanged)

Same principle as LSTM: an additive path where ∂h_t/∂h_{t-1} can be ≈ 1
No matrix multiplication on the carry-forward path — no exponential decay

GRU vs LSTM: The Key Differences

AspectLSTMGRU
States Two: h_t (hidden) + C_t (cell) One: h_t only
Gates Three: forget, input, output Two: reset, update
Parameters 4 × (hidden² + hidden×input + hidden) 3 × (hidden² + hidden×input + hidden)
Example (h=256, x=128) ~330k parameters ~248k parameters (~25% fewer)
Gradient highway Through cell state C_t (explicit separate path) Through update gate (1−z_t carries h forward)
Exposed memory Cell state C_t is private; h_t is the output h_t serves both roles — simpler but less separation
Performance ~ Better on long sequences, complex tasks ~ Better on small datasets, faster training
Still used in Speech recognition, music generation, long documents Time series, smaller models, faster inference
📌 Which Should You Use?

The honest answer: benchmark both on your task. The theoretical advantage of LSTM (explicit separation of long and short-term memory) matters on tasks with very long-range dependencies — paragraph-level coreference resolution, long music sequences, complex structured prediction. On shorter sequences (<100 tokens) and smaller datasets, GRU often matches or beats LSTM with fewer parameters and faster training.

For new projects in 2026: if you need an RNN-style model at all (streaming data, edge deployment, constant memory per step), try GRU first. Its simplicity is an advantage when data is limited. If you see a gap vs LSTM on your validation set, switch. Both are far outperformed by transformers when you have enough data and compute — which is most modern NLP tasks.

09 — Code: LSTM and GRU from Scratch

Complete implementations of both architectures — every gate written explicitly, a stacked bidirectional variant, and a training comparison against vanilla RNN on a long-range dependency task.

import torch
import torch.nn as nn
import math

# ─────────────────────────────────────────────────────────────────────────
# 1. LSTM Cell — all four equations explicit, every gate visible
# ─────────────────────────────────────────────────────────────────────────
class LSTMCellManual(nn.Module):
    def __init__(self, input_size, hidden_size):
        super().__init__()
        self.hidden_size = hidden_size
        # One combined weight matrix for all 4 gates — efficient but readable when split
        # W: (4*hidden, input+hidden)  —  [forget | input | candidate | output]
        self.W = nn.Linear(input_size + hidden_size, 4 * hidden_size)
        self._init_weights()

    def _init_weights(self):
        # Orthogonal init for the recurrent part, uniform for input part
        nn.init.orthogonal_(self.W.weight[:, :self.hidden_size])
        nn.init.xavier_uniform_(self.W.weight[:, self.hidden_size:])
        # Forget gate bias to 1.0 — encourages remembering at initialisation
        nn.init.ones_(self.W.bias[:self.hidden_size])

    def forward(self, x_t, h_prev, c_prev):
        """
        x_t:    (batch, input_size)
        h_prev: (batch, hidden_size)
        c_prev: (batch, hidden_size)
        returns: h_t, c_t — each (batch, hidden_size)
        """
        combined = torch.cat([h_prev, x_t], dim=1)   # (batch, hidden+input)
        gates    = self.W(combined)                 # (batch, 4*hidden)

        # Split into 4 gates along dim=1
        f, i, g, o = gates.chunk(4, dim=1)

        f = torch.sigmoid(f)       # forget gate  → (0, 1)
        i = torch.sigmoid(i)       # input gate   → (0, 1)
        g = torch.tanh(g)          # candidate    → (-1, 1)
        o = torch.sigmoid(o)       # output gate  → (0, 1)

        c_t = f * c_prev + i * g   # cell update — the gradient highway (addition!)
        h_t = o * torch.tanh(c_t)  # hidden state — filtered cell
        return h_t, c_t


# ─────────────────────────────────────────────────────────────────────────
# 2. GRU Cell — two gates, one state
# ─────────────────────────────────────────────────────────────────────────
class GRUCellManual(nn.Module):
    def __init__(self, input_size, hidden_size):
        super().__init__()
        self.hidden_size = hidden_size
        # Reset + update gates share one projection from [h, x]
        self.W_rz = nn.Linear(input_size + hidden_size, 2 * hidden_size)
        # Candidate uses reset-gated h_{t-1}, so separate projection
        self.W_n_h = nn.Linear(hidden_size, hidden_size, bias=False)
        self.W_n_x = nn.Linear(input_size, hidden_size)

    def forward(self, x_t, h_prev):
        """
        x_t:    (batch, input_size)
        h_prev: (batch, hidden_size)
        returns: h_t (batch, hidden_size)
        """
        combined = torch.cat([h_prev, x_t], dim=1)
        rz       = torch.sigmoid(self.W_rz(combined))   # (batch, 2*hidden)
        r, z     = rz.chunk(2, dim=1)                    # reset, update gates

        # Candidate: reset gate filters how much h_{t-1} contributes
        n = torch.tanh(self.W_n_h(r * h_prev) + self.W_n_x(x_t))

        # Interpolate: z=1 means full update, z=0 means carry forward
        h_t = (1 - z) * h_prev + z * n
        return h_t


# ─────────────────────────────────────────────────────────────────────────
# 3. Full sequence processors (many-to-many and many-to-one)
# ─────────────────────────────────────────────────────────────────────────
class LSTMSequence(nn.Module):
    def __init__(self, input_size, hidden_size, output_size, num_layers=1):
        super().__init__()
        self.hidden_size = hidden_size
        self.num_layers  = num_layers
        # Stack of LSTM cells — each layer takes previous layer's h sequence as input
        self.cells = nn.ModuleList([
            LSTMCellManual(input_size if l == 0 else hidden_size, hidden_size)
            for l in range(num_layers)
        ])
        self.fc = nn.Linear(hidden_size, output_size)

    def forward(self, x):
        """x: (batch, seq_len, input_size)"""
        B, T, _ = x.shape
        # Initialise all layers' h and c to zeros
        h = [torch.zeros(B, self.hidden_size) for _ in range(self.num_layers)]
        c = [torch.zeros(B, self.hidden_size) for _ in range(self.num_layers)]

        for t in range(T):
            layer_input = x[:, t, :]
            for l, cell in enumerate(self.cells):
                h[l], c[l] = cell(layer_input, h[l], c[l])
                layer_input = h[l]   # next layer reads this layer's h

        return self.fc(h[-1])   # classify from final layer's final hidden state


class GRUSequence(nn.Module):
    def __init__(self, input_size, hidden_size, output_size):
        super().__init__()
        self.hidden_size = hidden_size
        self.cell = GRUCellManual(input_size, hidden_size)
        self.fc   = nn.Linear(hidden_size, output_size)

    def forward(self, x):
        B, T, _ = x.shape
        h = torch.zeros(B, self.hidden_size)
        for t in range(T):
            h = self.cell(x[:, t, :], h)
        return self.fc(h)


# ─────────────────────────────────────────────────────────────────────────
# 4. Long-range dependency task — first token determines label
#    The model must remember token[0] across the full sequence
# ─────────────────────────────────────────────────────────────────────────
def make_long_range_batch(batch_size, seq_len, input_size):
    """Label = sign of first token's first dimension. Model must remember it."""
    x = torch.randn(batch_size, seq_len, input_size)
    y = (x[:, 0, 0] > 0).long()   # label from FIRST token only
    return x, y

def train_compare(seq_len=40, steps=300):
    """Train RNN, LSTM, GRU on a long-range task and compare final accuracy."""
    input_sz, hidden_sz = 8, 32

    from rnn_day20 import SimpleRNN   # conceptual import — use nn.RNN below instead
    models = {
        "RNN ":  nn.Sequential(nn.RNN(input_sz, hidden_sz, batch_first=True)),
        "LSTM": LSTMSequence(input_sz, hidden_sz, output_size=2),
        "GRU ": GRUSequence(input_sz, hidden_sz, output_size=2),
    }

    results = {}
    for name, model in models.items():
        if name.strip() == "RNN":   # wrap nn.RNN output
            continue
        optim = torch.optim.Adam(model.parameters(), lr=1e-3)
        correct = 0
        for step in range(steps):
            x, y = make_long_range_batch(64, seq_len, input_sz)
            optim.zero_grad()
            out  = model(x)
            loss = nn.CrossEntropyLoss()(out, y)
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optim.step()
            if step == steps - 1:
                correct = (out.argmax(1) == y).float().mean().item()
        results[name] = correct
    return results

out_lstm = lstm_model(x)

10 — Where LSTM and GRU Still Win in 2026

Transformers dominate NLP. But LSTM and GRU are not dead — they remain the right tool in specific, well-defined scenarios:

Use caseWhy LSTM/GRU winsExamples in production
Streaming / online inference Process one token at a time with constant O(1) memory — no need to store the full KV cache. Transformers require O(n) memory per token generated. Real-time speech recognition, live captioning, financial tick-level models
Very long sequences For sequences of 10k+ steps, transformer attention is O(n²) — prohibitive. LSTM is O(n). State-space models (Mamba, S4) are the modern heir, but LSTM still competes on certain tasks. Genomic sequence modelling, EEG/ECG signal analysis, long audio
Edge / embedded deployment A small LSTM (hidden=64) has ~50k parameters and runs on a microcontroller. The smallest practical transformer is orders of magnitude larger. Keyword spotting on-device, predictive text on low-power hardware
Small dataset regime Transformers need vast data to shine — their attention mechanism has no inductive bias toward local patterns. LSTM's sequential inductive bias helps generalise from few examples. Medical time series (limited patient data), niche domain classification
Hybrid architectures Some models combine transformer and RNN components — transformer for global context, LSTM for local sequential refinement. Longformer variants, some speech synthesis models
⚠️ Where LSTM/GRU Cannot Compete

For any task where you have: (a) enough data (>10k examples), (b) sequences under ~2k tokens, and (c) enough compute to run a transformer — transformers win, consistently. BERT-base outperforms BiLSTM on NER with the same data. GPT-style models outperform LSTM language models on every standard benchmark. The transformer's ability to directly connect any two tokens (no gradient chain at all) is a fundamentally better architecture for learning long-range dependencies. LSTM was the best solution for 20 years. Transformer made it obsolete for most applications.

11 — The Remaining Bottleneck: Why Even LSTM Isn't Enough for Translation

LSTM and GRU solve the vanishing gradient. But there's a second problem they don't solve — the information bottleneck in seq2seq models.

Recall the seq2seq architecture from Day 20: an encoder reads the full source sentence and compresses it into a single context vector (the final hidden state h_T). A decoder then generates the target sentence conditioned on this one vector.

Source: "The cat that sat on the mat in the garden is sleeping" │ ▼ Encoder LSTM (reads all 11 words) │ ▼ Context vector h_T ← ONE vector, fixed size (e.g. 512 dimensions) ← Must encode ALL 11 words, their relationships, ← the subject "cat", the modifier "in the garden", ← the predicate "is sleeping" ← Everything compressed into 512 numbers │ ▼ Decoder LSTM (generates translation from this one vector) │ ▼ "Die Katze, die auf der Matte im Garten saß, schläft"

For short sentences, this works well. For longer sentences (20+ words), the fixed-size context vector becomes a bottleneck — it simply cannot losslessly encode everything. Google's GNMT (2016) using 8-layer stacked LSTMs achieved impressive results but still degraded on long sentences.

The fix — introduced by Bahdanau et al. in 2015 — was attention: instead of compressing everything into one vector, let the decoder dynamically look at all encoder hidden states at each decoding step, and weight them by relevance. This is the direct ancestor of the transformer's self-attention. Day 24 covers it in full.

🔗 The Chain of Progress

RNN (1986) → identified sequential processing, struggled with long-range deps
LSTM (1997) → solved vanishing gradient via cell state + gates
GRU (2014) → simplified LSTM to 2 gates, comparable performance
Attention (2015) → solved the seq2seq bottleneck, introduced direct token connections
Transformer (2017) → removed recurrence entirely, O(n²) attention over all tokens

Each step solved one specific, identified problem with the previous architecture. The transformer didn't appear from nowhere — it's the culmination of 30 years of precisely diagnosing and fixing failure modes.

12 — A Thought to Sit With

💬 For You to Consider

The LSTM's forget gate starts with a bias of 1.0 — a deliberate initialisation choice that says "start by remembering everything, then learn what to forget." This is the opposite of starting from a blank slate. The model begins life as a perfect memory system and gradually learns selective forgetting.

Human memory works differently — we forget by default and remember by rehearsal. An LSTM's default is to remember until explicitly told to forget. Is this a better prior for language? And what does it suggest about how the model builds up a representation of a sentence — as a gradually pruned full memory, rather than a gradually constructed selective one?

🧠 Check Your Understanding
1. What is the gradient highway and why does it solve the vanishing gradient problem mathematically? Be precise.
The gradient highway is the cell state C_t in the LSTM — a memory variable updated by addition: C_t = f_t ⊙ C_{t-1} + i_t ⊙ g_t. The mathematical key is in the gradient ∂C_T/∂C_1 = ∏(t=2 to T) f_t, the product of forget gate values. Compare to the RNN: ∂h_T/∂h_1 = ∏(t=2 to T) diag(sech²(·)) · W_h — a product involving W_h multiplied by itself T-1 times. The RNN gradient involves a matrix product that shrinks exponentially when ||W_h||<1. The LSTM cell gradient only involves the forget gate values, which are: (a) scalar multiplications, not matrix multiplications; (b) learned by the model — it can set f_t ≈ 1 for dimensions it wants to preserve. When f_t = 1 exactly, ∂C_t/∂C_{t-1} = 1 — gradient passes through that step unchanged. The model learns to keep forget gates near 1 for important long-range dependencies and near 0 for information that should be discarded. The gradient path is controlled by the model, not forced to decay by the architecture.
2. Walk through what each LSTM gate does for the sentence: "The bank by the river is flooded." What might f_t, i_t, and o_t do at the word "flooded"?
At the word "flooded" (the final content word): Forget gate f_t: likely stays high (~0.9) for dimensions encoding the subject "bank" and its modifier "by the river" — these are still grammatically relevant as the model needs to complete the sentence prediction. May partially open (lower f_t) for dimensions encoding positional/function word information no longer needed. Input gate i_t: likely high for the "flooded" token — it is the predicate, the most semantically important word for predicting the end-of-sentence and for any downstream classification task (e.g., classifying this as a weather/flooding topic). The candidate g_t simultaneously proposes what "flooded" encodes (negative sentiment, water event, current tense, predicate role). Cell update writes this: C_t receives old memory × f_t plus "flooded" encoding × i_t. Output gate o_t: the most interesting. At "flooded", if the model is doing next-word prediction, it might expose the word's semantic content strongly (predicting "." or downstream tokens). In a classification task, it might expose the sentiment and topic features while keeping structural memory private. The key point: none of this is hardcoded. The model learns these gate values entirely from the training objective. What we describe is what trained LSTMs have been shown to learn via probing experiments and attention analysis.
3. The GRU has no explicit cell state. How does it still maintain a gradient highway? What is the equivalent mechanism?
The GRU's gradient highway is through the update equation: h_t = (1 − z_t) ⊙ h_{t-1} + z_t ⊙ ñ_t. The gradient ∂h_t/∂h_{t-1} has two components: (1) The direct path (1 − z_t): when z_t ≈ 0, this is ≈ 1 — h_{t-1} is passed forward with gradient ≈ 1. This is analogous to the LSTM cell state when f_t ≈ 1. (2) The ñ_t path via the reset gate: ñ_t = tanh(W_n_h · (r_t ⊙ h_{t-1}) + W_n_x · x_t), so ∂ñ_t/∂h_{t-1} involves r_t and tanh derivative — this part can still vanish. But the first component (1 − z_t carries h_{t-1} forward) is the highway. When the model sets z_t close to 0, old information is passed forward with gradient ≈ 1 regardless of sequence length. The difference from LSTM: in LSTM, the cell state C_t is a separate, dedicated gradient highway explicitly decoupled from the output h_t. In GRU, the hidden state h_t serves both roles — it is both the carry-forward memory and the output. This means the GRU cannot independently control "how much to remember" and "what to expose to the next layer" the way LSTM can via o_t. For most tasks this doesn't matter, but for tasks requiring long-term memory and precise output filtering (e.g., machine translation with very different source/target structures), LSTM's explicit separation may help.
4. What is the forget gate bias initialisation trick? Why does it help, and what is the risk of not doing it?
The trick: initialise the forget gate bias b_f to 1.0 (or sometimes between 1 and 4), rather than 0. Effect: at initialisation, the forget gate f_t = σ(W_f·z + b_f) ≈ σ(1.0) ≈ 0.73 for zero inputs, meaning the cell state is retained 73% by default from the start. Why it helps: at the beginning of training, weight matrices are near zero (or random), so gates would default to σ(0) = 0.5 with b_f=0. A forget gate of 0.5 means the cell state halves every step — at sequence length 50, the initial cell state contributes 0.5^50 ≈ 10^-15 to the current cell, effectively zero. Training would then have to learn to increase the forget gate from 0.5 while simultaneously learning all other weights — a harder optimisation landscape. With b_f=1.0, the model starts with a near-open forget gate and learns to selectively close it. It's easier to learn "forget these specific things" than to learn "remember anything at all" from scratch. Risk of not doing it: slow convergence on tasks requiring long-range memory. The gradient highway is theoretically available but practically not used early in training, because the model hasn't learned to keep f_t near 1. This was identified by Gers et al. (2000) and is now standard practice, implemented in PyTorch's nn.LSTM via uniform initialisation of the forget gate bias.
5. You're building a real-time audio transcription system that must process microphone input at 16kHz with under 20ms latency. Should you use an LSTM, GRU, or transformer? Justify the architectural choice.
LSTM or GRU, not a transformer. Here's why: The transformer requires causal attention — at each timestep it attends to all previous tokens. At 16kHz audio with typical frame rates (~100 frames/second), 20ms latency means the model must produce output within 2 frames of receiving input. A transformer processing the full context to this point requires O(n²) attention computation — as the audio stream grows (minutes of audio = thousands of frames), latency grows with n, violating the 20ms budget. An LSTM processes each frame in O(1) with a fixed-size hidden state — processing the 1000th frame takes exactly the same time as the 1st frame. The memory footprint is fixed (just h_t and C_t, e.g. 512 floats each) regardless of stream length. Transformers also require seeing the "full context" before producing output — for real-time streaming, this means either accepting stale outputs (latency grows) or using a sliding window (loses long-range context). LSTM's natural causal processing (it has seen exactly tokens 1 through t when processing token t) is the right inductive bias here. In practice: real production systems like DeepSpeech (Mozilla) and streaming ASR in Google's on-device models use RNN/LSTM-based architectures. Whisper (OpenAI) is transformer-based but processes audio in fixed 30-second chunks — not real-time streaming. GRU over LSTM: with 20ms latency requirements, the 25% parameter reduction of GRU translates to ~25% faster inference, which may be the difference between meeting the latency budget or not. Use GRU unless you see a significant accuracy gap on your validation set.
⚡ Quick Recap — LSTM & GRU
  1. The root cause: RNN gradients vanish because ∂h_T/∂h_1 involves W_h multiplied T−1 times. If ||W_h||<1, this product → 0 exponentially. The fix: create a path where ∂state_T/∂state_1 does not involve repeated matrix multiplication.
  2. LSTM cell state: C_t is updated by addition — C_t = f_t ⊙ C_{t-1} + i_t ⊙ g_t. The gradient ∂C_T/∂C_1 = ∏ f_t (product of forget gate values), not a matrix product. When f_t ≈ 1, gradients flow back unchanged across hundreds of steps.
  3. Four LSTM equations: forget gate f_t (what to erase from C), input gate i_t and candidate g_t (what to write to C), cell update C_t = f_t⊙C_{t-1} + i_t⊙g_t (additive highway), output gate o_t and hidden h_t = o_t⊙tanh(C_t) (what to expose).
  4. Gate operations: sigmoid for gates (outputs 0→1, acts as soft switch), tanh for candidates (outputs −1→1, bounded new content), element-wise multiplication (⊙) everywhere — no matrix products on the cell state path.
  5. Forget gate initialisation: b_f = 1.0 at initialisation → f_t ≈ 0.73 by default → model starts by remembering and learns to selectively forget. Critical for long-range learning early in training.
  6. GRU: two gates (reset r_t, update z_t) and one state h_t. Update: h_t = (1−z_t)⊙h_{t-1} + z_t⊙ñ_t. When z_t ≈ 0, h_{t-1} carries forward with gradient ≈ 1. ~25% fewer parameters than LSTM, comparable performance on most tasks.
  7. Key GRU vs LSTM difference: LSTM has an explicit private cell state C_t (long-term memory) and a public h_t (working memory). GRU uses h_t for both. LSTM can independently control memory vs output; GRU cannot.
  8. LSTM variants: peephole connections (gates see C_{t-1}), coupled forget/input gates, Bidirectional LSTM (forward + backward, concatenate h_t), stacked LSTM (multiple layers, lower layer's h becomes upper layer's input).
  9. Still wins over transformer: streaming/real-time inference (O(1) per step), very long sequences (>10k steps where O(n²) attention is prohibitive), edge deployment (tiny parameter count), small dataset regime (sequential inductive bias helps).
  10. Remaining bottleneck: seq2seq LSTMs still compress everything into one fixed-size context vector — information bottleneck on long sentences. Attention (2015) solved this by letting the decoder look at all encoder states. That insight became the transformer. Day 24.
🎯 Interview Questions
  1. What problem does the LSTM's cell state solve, and how?
    The cell state solves the vanishing gradient by providing an additive recurrence. C_t = f_t⊙C_{t-1} + i_t⊙g_t uses addition, not matrix multiplication. The gradient ∂C_T/∂C_1 = ∏f_t — a product of learned scalar values that the model can keep near 1 for dimensions it wants to remember, maintaining gradient signal across hundreds of steps.
  2. Explain the forget, input, and output gates in plain English.
    Forget gate: "How much of what I already remember should I keep?" — erases irrelevant information at context boundaries. Input gate: "How much of this new token matters?" — controls what gets written to long-term memory. Output gate: "What part of my memory is relevant right now?" — filters the full cell state to produce the hidden state exposed to the next layer. Together they implement read, write, and erase operations on a differentiable memory.
  3. How is GRU different from LSTM? When would you prefer each?
    GRU collapses LSTM's three gates to two (reset and update) and eliminates the separate cell state — 25% fewer parameters, simpler implementation. The hidden state serves both as memory and output. Prefer GRU when training speed or parameter efficiency matters, or when datasets are small. Prefer LSTM when tasks require explicit separation of long-term memory (cell) and short-term output (hidden) — typically complex seq2seq tasks or very long sequences.
  4. What is the information bottleneck in seq2seq LSTM models and what solved it?
    In seq2seq, the encoder compresses the entire source sequence into a single fixed-size vector (final h_T). For long sentences, this vector cannot losslessly encode all relevant information — long-distance dependencies between source and target words are lost. Bahdanau attention (2015) solved this by letting the decoder dynamically attend to all encoder hidden states at each decoding step, weighting them by relevance to the current decoding position. This eliminates the fixed-size bottleneck and became the conceptual basis for transformer self-attention.
